# dev_landing — Portfolio API

Backend для лендинга-портфолио с формой обратной связи, AI-анализом обращений и автоответами.
> **Демо:** К сожалению, не удалось развернуть на публичном хостинге из-за ограничений инфраструктуры (Ollama требует значительных ресурсов RAM, а бесплатные тарифы Render/Railway их не предоставляют). Проект полностью работоспособен при локальном запуске — см. инструкцию ниже.

---

## ✨ Возможности

- 📬 **Приём контактных обращений** с полной валидацией данных
- 🤖 **AI-анализ обращений** — определение тональности, категории, приоритета
- 💬 **Автоматическая генерация ответа** через локальную LLM (Qwen2.5)
- 📧 **Email-уведомления** — владельцу сайта и автоответ пользователю
- 🛡️ **Rate limiting** — защита от спама (5 запросов в час на IP)
- 📊 **Метрики и статистика** по всем запросам
- 📝 **Логирование** всех операций в файл с ротацией
- 📚 **Swagger/OpenAPI документация** с возможностью тестирования
- 🔄 **Graceful fallback** — если AI недоступен, сервис продолжает работать

## Как запустить проект

### Требования

- Python 3.9+
- [Ollama](https://ollama.com/download) (для AI-функций)
- Git

### 1. Клонирование репозитория

```bash
git clone https://github.com/eargosha/portfolio-api.git
cd portfolio-api
```

### 2. Создание виртуального окружения

**Windows:**
```bash
python -m venv .venv
.venv\Scripts\activate
```

**Linux/Mac:**
```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 3. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 4. Настройка Ollama (для AI-функций)

```bash
# Установите Ollama с официального сайта: https://ollama.com/download

# Скачайте модель Qwen2.5 (3 миллиарда параметров, ~2 ГБ)
ollama pull qwen2.5:3b

# Проверьте, что модель установлена
ollama list

# Запустите Ollama (обычно работает как служба, но можно вручную)
ollama serve
```

> 💡 Если не хотите использовать AI — просто установите `AI_ENABLED=False` в `.env`. Сервис будет работать через rule-based fallback.

### 5. Настройка переменных окружения

Создайте файл `.env` в корне проекта (пример в `.env.example`):

### 6. Запуск приложения

```bash
python app.py
```

Приложение запустится на `http://localhost:5000`

### 7. Проверка

- Лендинг: `http://localhost:5000/`
- Swagger UI: `http://localhost:5000/api/docs`
- Health check: `http://localhost:5000/api/health`

```
├── instance/                  # SQLite база данных
├── .env                       # Переменные окружения (не в git!)
├── .gitignore
├── config.py                  # Конфигурация приложения
├── app.py                     # Точка входа, фабрика приложения
└── requirements.txt
```

### Почему такой выбор?

1. **Flask вместо FastAPI/Django** - для тестового задания важна простота и контроль. Flask даёт понять, как всё работает "под капотом".
2. **SQLite вместо PostgreSQL** - для демо достаточно, не требует настройки сервера. Код легко переключается на PostgreSQL через `DATABASE_URL`.
3. **Ollama вместо OpenAI API** - бесплатно, работает без интернета, не зависит от внешних сервисов и санкций.
4. **Файловое хранилище для rate limit** - избыточно использовать Redis для 1 IP-адреса в тестовом задании.
5. **Flask-RESTX** - автоматическая Swagger-документация без написания OpenAPI-спецификации вручную.

## 📡 API Endpoints

### `POST /api/contact`

Отправка контактного обращения.

**Request:**

```http
POST /api/contact
Content-Type: application/json

{
  "name": "Иван Петров",
  "email": "ivan@example.com",
  "phone": "+7 999 123-45-67",
  "message": "Здравствуйте! Хочу предложить вам вакансию Flutter-разработчика. Зарплата от 200к."
}
```

**Response (201 Created):**

```json
{
  "success": true,
  "contact_id": 42,
  "ai_analysis": {
    "sentiment": "positive",
    "category": "job_offer",
    "priority": "high",
    "ai_used": true,
    "ai_fallback": false
  },
  "email_status": {
    "owner_sent": true,
    "user_sent": true
  },
  "message": "Обращение успешно отправлено! Мы свяжемся с вами в ближайшее время."
}
```

**Ошибки:**

- `400 Bad Request` — ошибка валидации (некорректные данные)
- `429 Too Many Requests` — превышен лимит запросов
- `500 Internal Server Error` — внутренняя ошибка

### `GET /api/health`

Проверка работоспособности сервиса.

**Response (200 OK):**

```json
{
  "status": "healthy",

}
```

## 🤖 AI-интеграция

### Что делает AI

- **Анализ тональности** — определяет, позитивное/нейтральное/негативное обращение
- **Классификация типа запроса** — `job_offer` / `collaboration` / `question` / `feedback` / `other`
- **Определение приоритета** — `high` / `medium` / `low`
- **Генерация автоответа** — вежливый ответ от лица разработчика

### Как реализовано

Используется локальная модель Qwen2.5:3b через Ollama. Процесс состоит из двух запросов:

**Запрос #1 — Классификация:**

```
Промпт: "Ты — AI-ассистент, анализирующий обращения с сайта-портфолио...
Проанализируй сообщение и верни ТОЛЬКО валидный JSON объект...
От кого: {name}
Сообщение: {message}
Верни ТОЛЬКО JSON объект"

Ответ: {"sentiment": "positive", "category": "job_offer", "priority": "high"}
```

**Запрос #2 — Генерация ответа:**

```
Промпт: "Ты — AI-ассистент разработчика из России...
Напиши ВЕЖЛИВЫЙ, ПРОФЕССИОНАЛЬНЫЙ ОТВЕТ на русском языке от лица разработчика...
Верни ТОЛЬКО валидный JSON объект в формате:
{"auto_response": "твой полный текст ответа"}"

Ответ: {"auto_response": "Здравствуйте, Иван! Спасибо за интересное предложение..."}
```

### Graceful Fallback

Если Ollama недоступен (сервис не запущен, ошибка, таймаут), автоматически включается rule-based анализ:

- **Тональность** — по словарям позитивных/негативных слов с проверкой границ слов (`\b`)
- **Категория** — по ключевым словам для каждой категории
- **Приоритет** — по словам "срочно", "вакансия" и т.д.
- **Ответ** — шаблонные тексты для каждой категории

Результат fallback:

```json
{
  "sentiment": "positive",
  "category": "job_offer",
  "priority": "high",
  "auto_response": "Здравствуйте, Иван! Спасибо за интерес к моему профилю...",
  "ai_used": false,
  "ai_fallback": true
}
```

Сервис всегда возвращает результат и никогда не падает из-за недоступности AI.

### Прогрев модели

При старте приложения отправляется тестовый запрос в Ollama, чтобы загрузить модель в RAM. Это ускоряет последующие запросы с ~7 сек до ~2 сек.

## 🧠 Что сделано с помощью AI

### Сгенерировано с помощью AI

- HTML-шаблоны email-писем и фронтенд лендинга — стилизация, структура, адаптивность, использование Tailwind CSS
- Regex для валидации телефонов — паттерн для международных номеров
- SQLAlchemy модели — базовая структура модели Contact
- Часть промптов для Ollama — итеративно улучшались через диалог
- Тексты автоответов — шаблонные ответы для fallback-режима
- Объяснения архитектурных решений — помощь в формулировках
- Частично данный MD файл)))

### Промпты, которые использовал

- «Напиши HTML-шаблон email-уведомления с использованием inline CSS для Flask-Mail»
- «Создай html лендинг страницу с использованием TailwindCSS на основе предоставленных файлов резюме»
- «Сгенерируй regex для валидации российских и международных телефонов»
- «Создай промпт для LLM, который вернёт строго JSON с полями sentiment/category/priority»
- «Как реализовать graceful fallback в Python-сервисе?»

### Что пришлось исправлять вручную

- **Категоризация fallback** — слово «работа» совпадало с «работает» → переписал на проверку границ слов (`\b`)
- **Импорты в Flask** — циклические зависимости при инициализации расширений → вынес в `extensions.py`
- **Прогрев Ollama** — первый запрос был слишком медленным → добавил `warmup.py`
- **Дублирование логов** — Flask + Werkzeug создавали дубли → настроил `propagate = False`
- **JSON от LLM** — модель иногда возвращала markdown-обёртки → добавил regex-очистку

## 💾 Хранение данных

### Логи

- Место: `logs/app.log`
- Формат: `YYYY-MM-DD HH:MM:SS - logger_name - LEVEL - message`
- Ротация: `RotatingFileHandler` — 5 МБ на файл, 5 файлов в истории
- Что логируется:
  - Все входящие запросы (метод, путь, IP)
  - Все ответы (статус, время выполнения)
  - Все этапы обработки обращения (валидация, AI, email, БД)
  - Все ошибки с traceback

### Rate Limiting

- Место: `data/rate_limits.json`
- Алгоритм: скользящее окно (sliding window)
- Структура: `{ "ip_address": [timestamp1, timestamp2, ...] }`
- Потокобезопасность: `threading.Lock`
- Очистка: старые записи удаляются при каждом запросе

### Статистика (метрики)

- Место: `data/metrics.json`
- Что хранится:
  - Общее количество запросов
  - Успешные/неуспешные запросы
  - Разбивка по эндпоинтам
  - Разбивка по HTTP-статусам
  - Время последнего обновления

### База данных

- Место: `instance/portfolio.db` (SQLite)
- Таблица `contacts`:
  - Данные пользователя (name, email, phone, message)
  - AI-анализ (sentiment, category, priority, auto_response)
  - Флаги AI (ai_used, ai_fallback)
  - Метаданные (ip_address, user_agent, created_at)
  - Статус email (email_sent_to_owner, email_sent_to_user)
